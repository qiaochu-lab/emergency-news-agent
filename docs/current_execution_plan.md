# 当前执行计划（开发版）

## 1. 文档目的

本文件用于指导当前阶段的实际开发与迭代，不替代 `docs/pm_project_management.md`。重点回答：

- 现在已经做到哪里
- 下一阶段先做什么
- 每项任务的验收标准是什么
- 哪些地方可借鉴 `TrendRadar` 与 `follow-builders`

适用时间范围：2026-03 下旬起的 2-6 周迭代周期。

---

## 2. 当前基线判断

### 2.1 已完成基线

- 已完成从采集到周报输出的端到端流水线
- 已完成中间产物落盘，支持排查与复盘
- 已完成规则优先的去重、分类、评分
- 已完成基础 LLM 分析接入与 mock 兜底
- 已完成基础周报模板输出
- 已完成最小测试集，当前主流程可回归

### 2.2 当前短板

- 周报内容可读性仍偏模板化，分析层级不够清晰
- 不同来源类型共用相近分析逻辑，内容颗粒度不够细
- 当前多数来源仍停留在标题级或摘要级读取，缺少正文级原文观点提取
- 信源质量控制与运行记录机制还不够完善
- 调度能力偏基础，缺少日报/快报/告警等扩展模式
- 缺少明确的 backlog、风险登记、运行日志文档

### 2.3 当前阶段结论

当前项目不应再优先扩张功能面，而应进入“质量增强 + 产品化整理”阶段。

---

## 3. 迭代目标

## 3.1 本阶段总目标

将当前 MVP 从“能生成周报”提升到“可稳定运行、可读性更强、方便继续扩展为日报/告警”的版本。

## 3.2 本阶段成功标准

- 周报每次生成时，结构完整且重点判断不明显重复
- 高分事件的分析内容能区分“事件描述 / 本质判断 / 影响 / 建议”
- 能明确看出不同来源类型的处理差异
- 能记录每次运行的输入源、异常、输出结果
- 能给后续日报/快报模式预留调度与模板接口

## 4. 任务优先级

## 4.1 P0：本周必须推进

### P0-1 报告内容重构

目标：
提升周报可读性，减少重复句式和空泛判断。

任务：

- 将“事件分析”拆成固定槽位：`发生了什么 / 为什么重要 / 对谁有影响 / 接下来关注什么`
- 将“分领域深度观察”改为先写领域综述，再列关键进展
- 控制同一条事件在多个章节重复拷贝同一句分析
- 明确“本周核心观点”来自高分条目的再抽象，而不是模板平铺

验收标准：

- 样例周报中同一句重复出现次数明显下降
- “核心观点”“重点事件”“分领域观察”之间职责清晰
- 人工阅读 5 分钟内可抓住 3-5 条核心判断

借鉴来源：

- `follow-builders`：先做单项摘要，再做总装配
- `TrendRadar`：结构化 prompt 输出固定板块

### P0-2 分类型分析策略

目标：
让公告、新闻、论文、论坛类内容采用不同分析提示与摘要方式。

任务：

- 为 `official / news / paper / forum` 建立最小差异化提示词策略
- 公告类强调政策、部署、预算、标准
- 论文类强调方法、应用潜力、成熟度
- 新闻类强调事件、参与方、影响面
- 社区/论坛类强调观点信号、争议点、弱信号价值

验收标准：

- 同一批输入中，不同来源类型生成的语言风格和关注点有区分
- 论文类条目不再用新闻口吻写结论
- 论坛类条目不会被误写成正式公告

借鉴来源：

- `follow-builders` 的多 prompt 结构：推文、博客、播客分别总结

### P0-3 运行日志与已知问题文档

目标：
为后续持续运行建立最小运营机制。

任务：

- 新增 `docs/weekly_run_log.md`
- 新增 `docs/risk_register.md`
- 每次运行记录：时间、使用的 source registry、成功数、失败源、周报路径、人工备注
- 记录已知问题，如资源页误入、旧背景材料误入、摘要重复等

验收标准：

- 每次手动跑周报后可补 1 条日志
- 可快速查看最近失败源和重复出现的问题

### P0-4 正文抽取与三级读取策略落地

目标：
解决“知道有哪些来源，但不知道原文到底讲了什么”的问题。

任务：

- 按 `标题级 / 摘要级 / 正文级` 建立分层读取策略
- 新增 `enrich_fulltext` 阶段，对高分或入选正文条目补抓正文
- 优先适配官方公告、公司博客、重点行业媒体
- 为无法稳定抽取的站点准备通用正文读取 fallback
- 分析阶段优先消费正文，而不是只消费标题和简短摘要

验收标准：

- 随机抽取 10 条重点条目，至少 7 条能看出明确原文观点
- 重点事件分析不再只是“标题改写”
- 官方类和深度博客类条目的信息密度明显提升

参考文档：

- `docs/content_ingestion_strategy.md`

## 4.2 P1：下一阶段应该推进

### P1-1 信源分层管理

目标：
把信源从“一个列表”升级为“优先级明确的情报资产”。

任务：

- 将信源分为：核心源、观察源、实验源
- 为每类信源标记建议读取深度：标题级 / 摘要级 / 正文级
- 对每个源补充：价值说明、稳定性、维护优先级、替代源
- 增加轻量 source health 检查输出

验收标准：

- 新增信源时能明确知道它属于哪一层
- 失败时知道优先修哪个源

借鉴来源：

- `TrendRadar` 的数据源配置组织方式

### P1-2 调度预设扩展

目标：
为后续日报/快报/提醒模式预留结构。

任务：

- 在现有 scheduler 基础上增加预设概念
- 至少规划 3 类模式：`weekly_only`、`weekday_digest`、`alert_watch`
- 周报逻辑与未来日报逻辑共用采集与筛选，但允许不同输出模板

验收标准：

- 代码结构上能支持不同调度模式
- 不需要重写主流水线即可增加日报入口

借鉴来源：

- `TrendRadar` 的 timeline / preset 思路

### P1-3 证据链增强

目标：
让报告更像“情报产品”，而不是“泛泛摘要”。

任务：

- 每条重点事件保留明确来源链接
- 核心观点尽量能回溯到支撑条目
- 增加“为什么入选本周重点”的可见字段

验收标准：

- 报告中的重点事件都有明确来源
- 核心判断可回溯到 1 个或多个条目

借鉴来源：

- `follow-builders` 的强链接约束

## 5. 建议开发顺序

建议按以下顺序推进：

1. 先改报告结构和分析文案
2. 再补正文抽取和三级读取能力
3. 然后拆分不同来源类型的分析策略
4. 再补运行日志、风险登记、需求池
5. 再做信源分层和健康检查
6. 最后扩展调度预设，为日报/告警做准备

原因：

- 当前最大问题不是“没有功能”，而是“产出质量和可持续运营能力”
- 报告质量改善后，后续新增日报/推送才有意义

## 9. 一句话结论

当前最优路径是：

先把“报告更像报告”这件事做好，再把“系统更像产品”这件事补上，最后再扩到日报、提醒和投递。
