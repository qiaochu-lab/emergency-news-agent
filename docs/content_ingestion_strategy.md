# 内容读取与正文抽取策略

## 1. 目标

解决当前“拿到了信息源，但不知道原文到底讲了什么”的问题。

本策略的核心目标不是对所有来源都抓全文，而是建立一个分层读取机制：

- 低成本拿到足够多的候选信息
- 对重点条目补正文
- 让最终周报既有覆盖面，也有原文观点和完整逻辑

---

## 2. 当前问题

当前系统虽然已经完成采集、标准化、去重、分类、评分和分析主链路，但在“内容读取深度”上仍存在明显短板：

- RSS 类来源通常只拿到标题和 `description`
- API 类来源通常只拿到摘要、tweet 文本或短说明
- Web 类来源虽然尝试抓文章页，但还缺少稳定的正文抽取能力
- mock 模式只用于跑通流程，不代表真实正文分析效果

因此当前系统更接近“选题引擎”，还不是成熟的“原文观点提炼引擎”。

---

## 3. 三级读取策略

## 3.1 一级：标题级读取

适用场景：

- 热榜、X、论坛、短讯流
- 需要高覆盖、低成本、快速发现主题

读取内容：

- 标题
- 链接
- 时间
- 来源
- 少量摘要 / 简介

作用：

- 用于发现主题、做去重、初步分类和热度判断
- 不直接作为最终研报正文的唯一依据

对应参考：

- `TrendRadar` 的主流程主要就是这一层

## 3.2 二级：摘要级读取

适用场景：

- RSS feed
- arXiv / 论文摘要
- API 返回 summary 的内容
- 一般新闻页的 meta description

读取内容：

- 标题
- 来源
- 发布时间
- 摘要 / abstract / description

作用：

- 用于初步分析“讲了什么”
- 可支撑初步筛选和基础摘要
- 但通常还不足以支撑证券研报式“分析判断”

对应参考：

- 当前项目大部分来源实际处于这一层

## 3.3 三级：正文级读取

适用场景：

- 已进入高分区间的候选条目
- 官方公告、深度新闻、公司博客、技术博客、专题解读
- 最终要写入周报正文的重点条目

读取内容：

- 正文全文或尽可能完整的正文 Markdown / 纯文本
- 文章结构（小标题、段落）
- 原始观点、结论、数字、表述方式

作用：

- 用于提取“原文到底讲了什么”
- 用于生成证券周报式“事件概述 + 分析判断”
- 用于保留证据链和关键表述

对应参考：

- `TrendRadar` 的按需文章正文读取（Jina Reader）
- `follow-builders` 的中心化完整 feed 内容思路

---

## 4. 哪些来源必须抓正文

以下类型建议进入正文级读取：

### 4.1 官方公告类

例如：

- 工信部 / NIST / DARPA / 3GPP / 企业正式新闻稿

原因：

- 关键价值不在标题，而在正文中的政策表述、部署节奏、标准方向、资金与组织安排

### 4.2 深度新闻与行业解读类

例如：

- Light Reading
- Commercial UAV News
- The Batch
- Import AI

原因：

- 关键价值通常在正文的解释、背景、采访和结论，而不是标题

### 4.3 公司技术博客 / 产品博客

例如：

- OpenAI Blog
- Skydio Blog
- Ericsson Blog

原因：

- 需要看产品细节、能力边界、技术参数、应用场景

### 4.4 最终入选周报正文的高分条目

规则建议：

- `final_score >= 7.0` 且 `include_in_top_report = true`
- 或被人工指定为本周重点

原因：

- 最终成稿如果没有正文支撑，只会变成“列信息源”

---

## 5. 哪些来源可以不抓正文

以下类型不建议默认全文抓取：

- X / 社区论坛短帖
- 热榜流
- 资源页 / landing page / 目录页
- 背景介绍页
- 明显重复转载

处理方式：

- 保留标题级或摘要级信息
- 只作为弱信号、辅助验证或主题发现来源

---

## 6. 建议实现路径

## 6.1 第一阶段：保持现有采集层不变，增加“正文补抓”阶段

建议不要一开始就把所有 source adapter 重写，而是在 `score -> screen -> analyze` 之间插入一个新阶段：

`enrich_fulltext`

建议流程：

1. 先完成原有采集、标准化、去重、分类、评分
2. 对高分候选条目筛选出需要补正文的列表
3. 按来源类型调用正文抽取器
4. 将正文写回 enriched 数据层
5. 分析阶段优先使用 enriched full text

建议新增目录：

- `data/enriched/items.json`
- `src/emergency_intel/enrich/`

## 6.2 第二阶段：为重点站点增加站点级正文抽取规则

优先给以下站点做站点级适配：

- 官方公告站
- OpenAI / Ericsson / Skydio 等博客
- 重点行业媒体

抽取目标：

- 标题
- 发布时间
- 正文主区域
- 小标题
- 关键信息段

## 6.3 第三阶段：引入通用正文读取服务作为 fallback

可选方案：

- Jina Reader 类 URL -> Markdown 方案
- Readability / Trafilatura / Goose 类正文抽取
- 对无法稳定解析的网站使用远程正文读取服务兜底

原则：

- 本地规则优先
- 通用正文读取兜底
- 失败时保留摘要级输入，不要阻断主流程

---

## 7. 建议的技术实现手段

## 7.1 基础方案

- HTML 抽取：站点级 CSS / 结构规则
- meta 信息：`og:title`、`og:description`、`article:published_time`
- 文本清洗：去导航、去页脚、去相关推荐、去 cookie 横幅

## 7.2 通用正文抽取方案

建议优先评估：

- `trafilatura`
- `readability-lxml`
- `newspaper3k`
- 远程 URL -> Markdown 服务

目标：

- 输出可供 LLM 直接消费的干净正文文本
- 最好带小标题结构

## 7.3 成本控制方案

正文抽取不要全量执行，建议只对以下条目执行：

- 高分条目
- 被筛入周报正文的条目
- 人工指定关注条目
- 特定高价值来源的新文章

---

## 8. 分析阶段如何使用正文

有了正文后，分析输入应优先使用：

- 标题
- 来源
- 时间
- 摘要
- 正文全文 / 正文前若干段
- 关键小标题

分析任务应拆成两层：

### 8.1 原文观点提取

回答：

- 原文核心观点是什么
- 作者 / 机构真正强调了什么
- 提到了哪些数字、判断、结论

### 8.2 周报分析判断

回答：

- 为什么这件事值得进入本周周报
- 对产业链 / 技术方向 / 泛应急场景意味着什么
- 后续应关注哪些验证点

---

## 9. 验收标准

当正文抽取能力上线后，应至少满足以下标准：

- 随机抽取 10 条重点条目，至少 7 条能读出明确原文观点
- 重点事件不再只是“标题改写”
- 周报中的“分析判断”能明显看出基于正文内容，而不是基于标题猜测
- 官方公告和公司博客类条目的信息密度明显提升

---

## 10. 当前推荐执行顺序

1. 新增 `enrich_fulltext` 阶段
2. 只对高分 / 入选正文条目补正文
3. 优先适配官方公告和重点博客
4. 再引入通用正文读取兜底
5. 最后重写分析 prompt，让模型先提取原文观点再生成研报语言

---

## 11. 一句话结论

最终目标不是“抓到更多链接”，而是：

让系统先用低成本方式发现主题，再用正文级读取为重点条目补足原文逻辑，从而支撑真正像证券研报的输出。
