# 输出规则（面向 Agent）

> 目的：引导 Agent 专注于“对具体信息源的深度客观分析”，严禁生成空泛的宏观趋势、景气度炒作和没有支撑的主观预测。

---

## 1. 总体原则：摒弃“宏大叙事”，回归“情报拆解”

- **基于具体信源**：所有的分析必须基于一个清晰的信息源（政策文件、技术文档、招投标、官方公告或重要会议等）。如果信息来源不可靠或纯属营销文稿，必须剔除或降权评价。
- **严禁宏观趋势套话**：禁止使用诸如“周度景气向上”、“风口已至”、“爆发式增长”、“迎来黄金拐点”、“重磅利好”等券商研报常用炒作词汇。
- **事实优先，落地为王**：任何判断都必须能够解释这项技术或政策“在现实业务中究竟解决什么问题”。不强求全部套入到应急场景，纯 AI 最新进展、卫星宽带科技等方向就按其原生领域去分析其技术与产业链影响即可。

---

## 2. 结构强制约束

输出必须遵循 `docs/report_template.md` 的章节结构：

1. 情报摘要选编
2. 重点信息源深度拆解
3. 关键线索跟进：政策、公告与招投标情报
4. 风险与异常预警

 **绝对禁止执行的动作（Agent 红线）：**
- ❌ **禁止撰写**“周度景气度与板块涨跌表现”等类似章节。
- ❌ **禁止脱离信息源**大谈特谈技术趋势或行业风口。
- ❌ **禁止做任何主观二级市场投资推荐**（不写“建议关注方向”或“推荐标的”、“强烈推荐”等字眼）。
- ❌ **禁止在“信息溯源”段落掺杂主观臆断**，事实就是事实。

---

## 3. 各章节写作细则

### 3.1 情报摘要选编
- 仅选 3-5 条硬核结论。
- 强制句式：“[信息源主体] 本周表明 / 发布 [核心事实]，这导致 / 意味着 [落地影响]。”

### 3.2 重点信息源深度拆解（核心篇幅）
每条拆解必须严格分为“信息溯源”与“深度落地分析”。
- **信息溯源**要求极度客观：说清是谁在什么时间发的。如果有数据指标（如续航、载重、带宽等），必须原样引用。
- **深度落地分析**要求极度具体，必须强制覆盖：
  - **意图**：他们为什么发这个？
  - **场景与业务关联**：这项技术或政策的实际应用点和创新点是什么？（可以是纯通用 AI 能力提升、卫星技术突破，只在确属应急类新闻时，才聚焦到泛应急/救援通信场景上去，严禁生拉硬拽）。
  - **产业影响**：对现有设备商、开发者社区、运营商或技术体系等有什么实际改动？
  - 如果信息本身价值有限，必须客观写出其局限性，不要为了凑字数强行拔高。

### 3.3 关键线索跟进
- 只收录干货公告、招标或文件。带一句话短评即可。如果本周没有这一类高质量信息，可以留空或精简，宁缺毋滥。

### 3.4 风险与异常预警
- 必须是本周情报中真正显现的“坏消息”或“慢消息”。
- 不要编造类似“技术发展不及预期”的套话，必须写具体现象，例如：“本周 X 公司的演示暴露了该方案在复杂地形下的通信盲区问题”。

---

## 4. 语气与词汇黑白名单

**✅ 推荐使用的词汇与句式（冷静、克制、事实驱动）：**
- 事实表明 / 数据显示 / 报告透露 / 文件明确规定
- 旨在解决 / 该路线验证了 / 补充了现有体系的短板
- 仍面临挑战 / 实际落地效果需观察 / 指标存在局限性
- 核心变量在于 / 主要瓶颈在于

**❌ 严禁使用的词汇与句式（营销、煽动、主观预测）：**
- 爆发式增长 / 千亿蓝海 / 黄金拐点 / 巨大红利
- 建议关注 / 强烈推荐 / 迎来风口 / 完美解决
- 我们认为 / 我们预测（即使预测也应替换为“从该文本推断”）

---

## 5. 最终自检清单
Agent 生成长文后，必须严格遵循防幻觉策略：
1. 我的摘要里是否有脱离具体信源的趋势判断？如果有，删掉。
2. 我是否在分析时使用了“风口”或“景气度”？如果有，改写。
3. 我对事件的分析，是否自然且客观？如果没有应急的相关度，我是否保持了在通信、AI、低空等它原生领域的专业分析而没有强行生编硬造？如果是，予以保留。
