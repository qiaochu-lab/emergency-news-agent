# 项目管理文档（产品经理视角）

## 一、项目概述

- 项目名称：泛应急新闻 Agent / Emergency Intelligence Weekly Report System
- 当前定位：一个面向决策者的周报型情报生产 MVP
- 当前形态：通过采集、标准化、去重、分类、评分、分析、成文等流水线，生成中文 Markdown 周报
- 当前阶段：MVP 已具备端到端跑通能力，正在从“能跑”走向“稳定可运营”
- 当前开发执行计划：见 `docs/current_execution_plan.md`

## 二、需求

### 1. 业务背景

泛应急相关信息分散在新闻、机构公告、论文、社区和公司博客中，人工筛选成本高、重复信息多、判断标准不统一。项目需要把多源信息加工成适合管理层阅读的周报，降低信息获取与判断成本。

### 2. 目标用户

- 决策者：希望快速理解本周重点事件、主线趋势和后续关注方向
- 研究/分析人员：希望从多源信息中快速定位高价值线索
- 项目运营者：希望每周稳定产出，不因单一来源失败而中断

### 3. 核心需求

- 每周稳定生成一份中文周报
- 对多源信息进行统一清洗与结构化
- 自动去重，减少同一事件多次进入报告
- 按 AI、无人机、通信、泛应急等主题分类
- 对事件进行重要性和热度评分，优先展示高信号内容
- 对重点条目补足正文级内容，避免最终报告只停留在标题改写和信息罗列
- 在模型可用时生成结构化分析，在模型不可用时保留可兜底输出
- 保留中间产物，方便复盘、排查和二次加工

### 4. MVP 范围

- 已纳入：
  - 手动运行采集与周报生成
  - 周期调度能力
  - 规则优先的去重、分类、评分
  - Provider 抽象下的结构化分析
  - Markdown 周报输出
- 暂不纳入：
  - Dashboard
  - 实时流式监控
  - 自动发现新信源
  - 复杂多 Agent 编排
  - 知识图谱

## 三、计划

### 1. 当前阶段判断

当前项目已经完成 MVP 主链路，重点应从“继续堆功能”转向“提升内容质量、信源质量和运营可控性”。

### 2. 阶段计划

#### 阶段 A：MVP 跑通与可演示

- 状态：已完成
- 目标：实现从采集到周报输出的端到端闭环

#### 阶段 B：内容质量与可运营性提升

- 状态：进行中
- 重点：
  - 增加更稳定、更高质量的信源
  - 建立标题级 / 摘要级 / 正文级的分层读取机制
  - 优化筛选逻辑，减少“资源页”“旧背景材料”等低价值内容进入报告
  - 提升周报可读性与中文表达质量
  - 建立固定的运行检查清单

#### 阶段 C：产品化与协同管理

- 状态：待开始
- 重点：
  - 增加版本记录与发布节奏
  - 建立需求池、优先级和验收标准
  - 形成面向运营的使用手册和异常处理手册
  - 视需要增加可视化页面或管理后台

### 3. 建议任务优先级

#### P0

- 保证每周周报稳定产出
- 维护高质量信源清单
- 为重点条目建立正文补抓能力
- 持续验证评分、筛选、分析逻辑是否符合预期

#### P1

- 提升报告内容密度与中文表达质量
- 为不同受众增加更清晰的摘要层级
- 增加更多真实场景下的测试样本

#### P2

- 增加 Dashboard 或检索界面
- 增加专题报告、日报或监控告警能力

## 四、目标

### 1. 产品目标

- 让用户在 5 分钟内理解本周泛应急领域的重点事件与趋势
- 将多源情报整理为统一、可复用、可追溯的报告资产
- 建立可每周持续运行的情报生产机制

### 2. 结果目标

- 每周稳定输出 1 份周报
- 报告中重点事件具备明确来源、主题归属和解释结论
- 报告中的重点事件应能体现原文核心观点，而不是仅复述标题
- 报告生成过程可复盘，失败可定位

### 3. 建议量化指标

- 产出稳定性：周报按计划生成成功率
- 内容有效性：进入 Top Signals 的条目中，高价值条目占比
- 噪音控制：被判定为低价值/资源页/过时背景材料的误入比例
- 正文覆盖率：进入周报正文的条目中，具备正文级内容的占比
- 处理效率：单次流水线执行时长
- 信源健康度：有效信源数量、连续失败信源数量

- `src/emergency_intel/enrich/`：重点条目正文补抓（建议新增）
- `src/emergency_intel/normalize/`：标准化
- `src/emergency_intel/dedup/`：去重
- `src/emergency_intel/classify/`：分类
- `src/emergency_intel/score/`：评分
- `src/emergency_intel/analyze/`：结构化分析与筛选
- `src/emergency_intel/report/`：周报生成
- `src/emergency_intel/pipeline.py`：端到端流水线编排
- `src/emergency_intel/scheduler.py`：调度时间计算

### 2. 运行脚本

- `scripts/run_collect_once.py`：单次采集
- `scripts/run_weekly_report.py`：生成周报
- `scripts/run_scheduler.py`：按计划循环运行

### 3. 数据与输出目录

- `data/source_registry.json`：主信源注册表，当前 22 个来源
- `data/source_registry_live_quick.json`：轻量信源注册表，当前 3 个来源
- `data/raw/`：原始采集结果
- `data/normalized/`：标准化结果
- `data/deduped/`：去重结果
- `data/classified/`：分类结果
- `data/scored/`：评分、筛选、分析结果
- `outputs/weekly/`：最终周报产物

### 4. 外部依赖与配置

- Python 3.9+
- `.env` / 环境变量配置
- 支持 `mock` 与 `openai_compatible` 两类模型提供方
- 可配置 X API Bearer Token
